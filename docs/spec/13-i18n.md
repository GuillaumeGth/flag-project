# 13 — Internationalisation (i18n)

## Vue d'ensemble

L'app est entièrement traduite via **i18next** + **react-i18next**. La langue est détectée automatiquement depuis le téléphone et peut être changée par l'utilisateur dans les Paramètres.

**Langues supportées :** Français (`fr`), Anglais (`en`), Espagnol (`es`), Portugais (`pt`), Allemand (`de`)
**Fallback :** `en` (si la langue du device n'est pas supportée)

---

## Règle absolue — zéro texte statique

> **Toute string visible par l'utilisateur doit passer par `t()`.**
> Il est interdit de créer du texte statique dans les composants ou écrans.

```tsx
// ❌ Interdit
<Text>Aucune conversation</Text>
<PremiumButton title="Envoyer" />
placeholder="Rechercher..."

// ✅ Obligatoire
<Text>{t('inbox.empty')}</Text>
<PremiumButton title={t('sendMessage.send')} />
placeholder={t('search.placeholder')}
```

Cela inclut :
- Les `<Text>` JSX
- Les props `title`, `placeholder`, `label`, `subtitle` des composants
- Les `message` et `confirmLabel` des `ConfirmDialog`
- Les options de listes (`label:` dans les tableaux de constantes)

---

## Utilisation dans un composant

```tsx
import { useTranslation } from 'react-i18next';

export default function MonComposant() {
  const { t } = useTranslation();

  return <Text>{t('inbox.title')}</Text>;
}
```

`useTranslation` ne peut s'utiliser que dans un composant React (pas dans un service, ni dans une constante module-level).

### Cas spécial : constantes module-level avec labels traduits

Si une constante (ex: options d'un filtre) utilise des labels traduits, elle doit être **déclarée à l'intérieur du composant** (ou via `useMemo`) — pas au niveau du module.

```tsx
// ❌ Ne compile pas / ne se met pas à jour lors du changement de langue
const OPTIONS = [{ label: 'Tous', value: 'all' }];

// ✅ Dans le composant
export default function MonComposant() {
  const { t } = useTranslation();
  const OPTIONS = [
    { label: t('mapFilter.all'), value: 'all' },
    { label: t('mapFilter.public'), value: 'public' },
  ];
}
```

## Ajouter une nouvelle clé

1. **Ajouter la clé dans `fr.ts`** (source de vérité)
2. **Ajouter la même clé dans `en.ts`, `es.ts`, `pt.ts`, `de.ts`**
3. **Utiliser `t('section.clé')` dans le composant**

> Le test `__tests__/services/i18n.test.ts` vérifie automatiquement que toutes les clés de `fr.ts` existent dans les autres locales — il échouera si une clé est oubliée.

## Ajouter une nouvelle langue

1. Créer `src/i18n/locales/xx.ts` en implémentant le type `Translations` importé de `fr.ts`
2. Ajouter la langue dans `src/i18n/index.ts` :
   ```ts
   import xx from './locales/xx';

   export const SUPPORTED_LANGUAGES = ['fr', 'en', 'es', 'pt', 'de', 'xx'] as const;
   export const LANGUAGE_LABELS = { ..., xx: 'Nom affiché' };

   // Dans initI18n :
   resources: { ..., xx: { translation: xx } }
   ```
3. Aucun rebuild natif requis.

## Initialisation

L'init est **asynchrone** — `App.tsx` attend sa résolution avant le premier render :

```ts
// App.tsx
const [i18nReady, setI18nReady] = useState(false);

useEffect(() => {
  initI18n().then(() => setI18nReady(true));
}, []);

if (!i18nReady) return null;
```

### Ordre de priorité de la langue au démarrage

```
AsyncStorage ('app_language') → valide ?
  ✅ oui → utiliser cette langue
  ❌ non → langue du device (expo-localization)
               → supportée ?
                    ✅ oui → utiliser
                    ❌ non → fallback 'en'
```

---

## Changement de langue (runtime)

Le changement est **instantané** — tous les composants React se re-rendent automatiquement via le contexte `react-i18next`.

```ts
import { i18next, saveLanguage } from '@/i18n';

await i18next.changeLanguage('es'); // re-render immédiat
await saveLanguage('es');           // persistance AsyncStorage
```

Le sélecteur de langue est accessible dans **Paramètres → Langue**.

## Tests

Fichier : `__tests__/services/i18n.test.ts`

| Suite | Ce qui est testé |
|-------|-----------------|
| Cohérence des locales | Toutes les clés de `fr.ts` existent dans `en`, `es`, `pt`, `de` — aucune valeur vide |
| `getDeviceLanguage` | Langues supportées, non supportées, tableau vide, `null` |
| `getInitialLanguage` | AsyncStorage valide, vide, valeur invalide, erreur silencieuse |
| `saveLanguage` | Écriture AsyncStorage, silence sur erreur |

Lancer les tests i18n :
```bash
npm test -- --testPathPatterns="i18n"
```

## Ce qui n'est PAS traduit

- Le contenu des messages des utilisateurs (jamais traduit — c'est du contenu, pas de l'UI)
- Les noms d'utilisateurs
- Les notifications push générées côté Supabase (triggers DB — hors scope)
- Les messages d'erreur techniques loggés via `reportError`
